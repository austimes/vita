VedaLang Core Abstraction Note: Role, Variant, Mode, Provider, Instance, Scope
=============================================================================

Audience
--------
Implementation developers working on:
- Facility + fleet primitives
- Compilation naming and dimensionality
- RES/graph viewer granularity + commodity collapse controls
- Reporting keys (Safeguard/ASRS-style facility outputs)

Context / motivation
--------------------
We were seeing UX/model confusion driven by mixing two different axes in a single
hierarchy:

1) TYPE semantics: what transformation/service is provided and how (tech + operating state)
2) OBJECT semantics: which concrete entity is providing it (named facility, or regional fleet)

Additionally, we observed duplicate service commodities that differed only by scope
(e.g. heat@res vs heat@com), where scope is a commodity-only tradability boundary.

This note defines the minimal, clear abstraction and how it maps to:
- compilation (process naming)
- RES viewer grouping (granularity)
- commodity scope-collapse toggles
- reporting outputs


1. Definitions (single source of truth)
---------------------------------------

1.1 TYPE axis (design-time)

Role
  "What service/transformation is being provided?"
  Examples: steel_primary_production, high_temp_heat, space_heat

Variant
  "One technology pathway that can satisfy a role."
  Examples (steel): bf_bof, dri_eaf
  Examples (heat): gas_boiler, heat_pump, kiln_standard, kiln_electric

Mode
  "An operating state within a variant (often fuel/config)."
  Examples: coal, retrofit_to_ng, retrofit_to_h2, bio_methane, grid_electric, h2_dri

Hard rule (important):
  role → variant → mode is a fixed type hierarchy.
  Modes always belong to a variant; variants always belong to a role.
  This remains true for both facilities and fleets.

1.2 OBJECT axis (runtime / reporting)

Provider
  A provider is a concrete object that hosts type choices over time.
  Providers are reporting entities and appear in outputs and RES views.

  Provider kinds:
    Facility provider (site-specific; always a single model region)
      Examples: facility.sgf_alumina_gladstone, facility.port_kembla_steel

    Fleet provider (regional “virtual provider” representing distributed stock)
      Examples: fleet.space_heat.VIC.residential, fleet.low_temp_heat.NSW.commercial

Constraint for current phase:
  Providers are single-role (1 provider → 1 role).
  (Later: multi-role providers can be introduced if/when needed.)

Instance
  A compiler-resolved TIMES/VEDA process symbol for a specific:
    provider × role × variant × mode × region × scope resolution.
  Instances are implementation/debug level; humans should prefer provider/variant/mode.

1.3 COMMODITY axis (markets)

Scope (commodity-only tradability boundary)
  Scope is NOT a process concept.
  Scope partitions commodity markets to represent non-tradability across categories.

  Example:
    heat produced in commercial buildings cannot satisfy residential heat demand,
    so compiler lowers a logical commodity "heat" into scoped commodities:
      heat@com, heat@res

Scope affects:
  - commodity symbols
  - market-clearing / balance constraints
  - trade connectivity

Scope must NOT affect:
  - role/variant/mode semantics
  - provider identity


2. Conceptual model (how things fit)
------------------------------------

A role is satisfied by one or more providers.
Each provider meets its role by choosing variants and modes over time.

Read this as:

  "Provider P meets Role R by selecting Variant V and operating in Mode M over time,
   consuming and producing physical commodities, subject to scope-partitioned markets."

This makes "facility vs variant" no longer ambiguous:
- Facility is an OBJECT (provider)
- Variant is a TYPE (technology pathway)
- Mode is a TYPE (operating state within a pathway)


3. Diagrams
-----------

3.1 Two-axis mental model (Type vs Object vs Commodity)

TYPE (design-time)                     OBJECT (runtime/reporting)       COMMODITY (markets)
-----------------                     --------------------------       --------------------
Role  ─┬─ Variant ─┬─ Mode             Provider (Facility or Fleet)     Commodity symbols
      │           │                    ├─ has region (facility: single)  ├─ physical commodities
      │           └─ Mode              └─ hosts choices over time         └─ scoped markets
      └─ Variant ─┬─ Mode                                                 (scope is commodity-only)
                  └─ Mode

Key: "scope" lives ONLY on the commodity side.

3.2 Provider state over time (single-role provider)

For a given provider P and role R:

time ─────────▶

P:R   chooses Variant V(t)
      chooses Mode M(t) within V(t)
      subject to:
        - allowed variants/modes
        - ramp / no-backsliding constraints (optional)
        - costs (CAPEX/OPEX), emissions pressure, etc.

3.3 How scope creates duplicate commodity nodes (and how collapse fixes it)

Logical service commodity:
  HEAT

Compiler lowering by scope:
  HEAT@res   (residential heat market)
  HEAT@com   (commercial heat market)

Graph symptom:
  Two distinct nodes that are the same "service" but not tradable.

Viewer fix:
  Provide a commodity aggregation toggle to merge @scope suffixes when desired.


4. Examples
-----------

4.1 Steelmaking (current simplification approach)

Role:
  steel_primary_production

Variants:
  bf_bof
  dri_eaf

Modes:
  bf_bof: coal, coal_plus_ccs (optional), bio_coke_blend (optional)
  dri_eaf: ng_dri, h2_dri

Providers:
  facility.port_kembla_steel   (region NSW)
  fleet.steel_primary.NSW      (optional “rest-of-region” provider)

Compilation emits instance processes like:
  FAC::port_kembla_steel::ROLE::steel_primary_production::VAR::bf_bof::MODE::coal
  FAC::port_kembla_steel::ROLE::steel_primary_production::VAR::dri_eaf::MODE::ng_dri
  FAC::port_kembla_steel::ROLE::steel_primary_production::VAR::dri_eaf::MODE::h2_dri

All produce the same output commodity (e.g., MAT:crude_steel) but consume different fuels/inputs.

4.2 Building space heat (scope-driven markets)

Role:
  space_heat

Variants:
  gas_boiler
  heat_pump

Modes (if needed):
  gas_boiler: ng, bio_methane
  heat_pump: grid_electric, renewable_electric

Providers:
  fleet.space_heat.VIC.residential
  fleet.space_heat.VIC.commercial

Scope lowering:
  NRG:heat@res
  NRG:heat@com

Key point:
  Providers/types do not change because of scope.
  Only commodity symbols/markets change.


5. Naming conventions (canonical)
---------------------------------

5.1 Provider IDs

Facility providers:
  facility.<facility_id>
  Examples:
    facility.sgf_alumina_gladstone
    facility.port_kembla_steel

Fleet providers:
  fleet.<role_id>.<region>.<segment?>
  Examples:
    fleet.space_heat.VIC.residential
    fleet.low_temp_heat.NSW.commercial

Notes:
- Facility is always single model region (per current constraint).
- Fleets may be one per region × segment where segment aligns to scope partitions.

5.2 Type IDs

Role IDs:
  lower_snake_case, nouns, stable.
  Examples: steel_primary_production, high_temp_heat, space_heat

Variant IDs:
  lower_snake_case, tech family.
  Examples: bf_bof, dri_eaf, kiln_standard, kiln_electric, gas_boiler, heat_pump

Mode IDs:
  lower_snake_case, operating state/fuel/config.
  Examples: coal, retrofit_to_ng, retrofit_to_h2, ng_dri, h2_dri, grid_electric

Hard rule:
  Mode IDs are scoped to a variant (same mode name may appear in multiple variants).

5.3 Instance process symbol (compiler output)

Canonical pattern:
  {PROVIDER_KIND}::{provider_id}::ROLE::{role_id}::VAR::{variant_id}::MODE::{mode_id}

Provider kind tokens:
  FAC  for facility providers
  FLT  for fleet providers

Examples:
  FAC::port_kembla_steel::ROLE::steel_primary_production::VAR::bf_bof::MODE::coal
  FLT::space_heat.VIC.residential::ROLE::space_heat::VAR::gas_boiler::MODE::ng

5.4 Commodity scope lowering

Logical commodity:
  NRG:heat

Lowered commodities:
  NRG:heat@res
  NRG:heat@com

Rule:
  The @scope suffix is reserved for tradability partitions only.
  Scopes should not be embedded into role/variant/mode/provider IDs.


6. RES viewer: grouping (granularity) and scope collapse (commodity toggle)
---------------------------------------------------------------------------

Problem:
"Granularity" was being used to solve two different issues:
- choosing the level of process/provider abstraction
- suppressing duplicate commodities created by scope

Solution:
Expose TWO independent controls.

6.1 Node grouping (process/provider abstraction)

Recommended grouping levels (in order):

1) Role
   - One node per role.

2) Provider  (DEFAULT when providers exist)
   - One node per provider (facility or fleet).
   - This aligns with reporting and stakeholder mental models.

3) Provider × Variant
   - Shows technology pathway choices per provider, without mode noise.

4) Provider × Variant × Mode
   - Shows operating/fuel switching per provider.

5) Instance (advanced/debug)
   - Shows compiled instance process symbols (region/scope-resolved).

6.2 Commodity aggregation (scope handling; independent toggle)

- scoped
    show distinct commodities per scope (heat@res, heat@com)

- collapse_scope
    merge across scope suffix; show logical commodity (heat)

- collapse_scope_region  (optional; future-proof)
    merge scope but preserve region splitting if/when multi-region providers appear

Recommended defaults:
- Grouping ≤ Provider×Variant:
    default commodity view = collapse_scope  (reduces clutter, improves comprehension)
- Grouping = Instance:
    default commodity view = scoped  (debug correctness)

Implementation note:
The collapse operation is purely a VIEW transformation in the query/render layer:
- do not mutate underlying model symbols
- maintain provenance (collapsed node has a list of underlying scoped nodes)


7. How this supports facilities + fleets + reporting (requirements mapping)
---------------------------------------------------------------------------

Given confirmed requirements:

1) Multi-role providers: NO (for now)
   - Simplifies provider identity and UI.

2) Facility is single region: YES
   - Provider → region is well-defined.

3) Scope is commodity-only: YES
   - Enforces separation of concerns.
   - Enables independent scope-collapse toggle.

4) Mixed fleets and facilities: YES
   - Both are providers; viewer grouping works for both.

5) Providers are reporting keys: YES
   - Provider is primary grouping dimension and label.
   - Outputs can map to Safeguard/ASRS facility reporting naturally.


8. Implementation suggestions (developer handoff)
-------------------------------------------------

8.1 VedaLang schema / AST
- Add a first-class Provider concept:
  - kind: facility | fleet
  - id
  - region (facility required; fleet includes region token)
  - role (single for now)
  - allowed variants + modes
  - baseline assignment in base year (variant+mode+capacity)

- Keep facility_template as authoring sugar:
  - expands to provider objects
  - survives as metadata only (debug panels), not as primary RES nodes.

8.2 Compiler
- Compile provider × role × variant × mode to processes (instances) with canonical names.
- Ensure mode is nested under variant in naming and in AST.

- Scope lowering:
  - apply during commodity symbol emission ONLY
  - keep type/object IDs free of scope tokens

8.3 RES viewer / query engine
- Implement node grouping as a projection:
  - parse canonical names into structured fields
  - group by selected fields (role/provider/variant/mode/instance)

- Implement commodity aggregation as a separate projection:
  - scoped: keep full commodity symbols
  - collapse_scope: strip @scope suffix in view layer and merge nodes
  - retain mapping of collapsed->underlying for explainability and tooltips

- Facets:
  - Populate from runtime parsed fields (role/provider/variant/mode/region/scope)
  - Avoid static facet lists that miss dynamically generated scopes


9. Acceptance criteria
----------------------

Model semantics:
- role→variant→mode hierarchy holds everywhere.
- providers (facility/fleet) are first-class and stable reporting keys.
- scope partitions markets only; does not leak into provider/type naming.

Viewer:
- grouping levels produce intuitive graphs for both facility-heavy and fleet-heavy models.
- scope-collapse removes duplicate service commodities without changing process grouping.
- instance view remains available for debugging.

Compilation:
- canonical naming is stable and parseable into structured fields.
- scope lowering only affects commodities.


Appendix A: Quick worked example (duplicate commodity collapse)
--------------------------------------------------------------

Suppose the compiler emits two scoped heat commodities:

  NRG:heat@res
  NRG:heat@com

In scoped view:
  two nodes appear in the graph and do not trade.

In collapse_scope view:
  they merge into a single displayed node NRG:heat with metadata:
    underlying = [NRG:heat@res, NRG:heat@com]

Edges:
- edges into/out of the collapsed node are merged by summing (or aggregating) weights.
- tooltips can show a breakdown by underlying scope commodity.
