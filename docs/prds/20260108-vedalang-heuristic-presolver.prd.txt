VedaLang Heuristic Pre-Solver (Compile-Time Solution Estimator)
PRD v1.0
Date: 2026-01-08
Owner: (TBD)
Status: Draft for implementation planning

1. Purpose
Develop a fast, deterministic “Heuristic Pre-Solver” that runs at (or before) VedaLang compile-time and produces:
- A feasible (or near-feasible) approximate plan for meeting demands over the model horizon, WITHOUT running the TIMES generator or full optimization.
- High-signal diagnostics that identify infeasibilities, missing links, bottlenecks, and likely drivers (cost/emissions/policy) before a full solve.
- Optional warm-start artifacts to seed downstream optimization.

This component must operate directly over the existing VedaLang Intermediate Representation (IR) schema. Where this PRD refers to “IR fields”, devs should map to existing fields or define adapters.

2. Goals
G1. Speed: Typical runs in 50 ms–5 s depending on model size and enabled features.
G2. Determinism: Same input IR → same outputs (no stochastic choices by default).
G3. Explainability: Every allocation/build decision produces a traceable rationale.
G4. Feasibility diagnostics: Identify unreachable demands, binding limits, and constraint conflicts early.
G5. Useful approximations: Produce reasonable directionality for technology selection, fuel shares, emissions, and costs.
G6. Tooling integration: Outputs are structured artifacts consumable by CLI, CI checks, IDE tooling, and downstream reporting.

3. Non-Goals
NG1. Not a replacement for TIMES optimization; not expected to approximate optimality under tight intertemporal or integer-like constraints.
NG2. Not a complete constraint system; only a curated subset of “compile-time enforceable” constraints is in scope.
NG3. Not a full power-system unit commitment, network flow with Kirchhoff’s laws, or detailed operational dispatch model.
NG4. Not required to support every TIMES feature on day one (e.g., endogenous learning, complex vintage accounting, advanced trade/transport formulations).

4. Primary Users and Use Cases
U1. Model builders: quick feedback while authoring VedaLang modules and scenarios.
U2. CI pipeline: fail-fast on structural issues before burning solver hours.
U3. Analysts: quick “sanity check” of scenario intent and direction of results.
U4. Agent workflows: give AI agents a quick evaluator to guide iterative edits.

Representative user stories:
- “As a modeler, I want to know if each demand is reachable from resources in every region/year/timeslice.”
- “As a dev, I want the compiler to surface missing efficiency/units/links as actionable diagnostics.”
- “As an analyst, I want an approximate emissions time series and implied carbon price needed to hit a cap.”
- “As CI, I want a hard failure if demand is unmet or if a policy constraint is provably infeasible.”

5. Product Concept
The Heuristic Pre-Solver is a compile-time module that:
1) Ingests existing IR.
2) Builds a simplified “solver graph” view (commodities/services nodes; process edges with coefficients and costs).
3) Runs one of two execution modes:
   - Baseline Greedy Mode (default): Cheapest-path routing + greedy allocation + capacity expansion.
   - Hybrid Flow Mode (optional): Solve a small min-cost flow / transportation problem for robustness (still pre-TIMES).
4) Optionally enforces selected policy constraints (emissions cap, renewable share, phase-outs) via shadow-price iteration.
5) Produces outputs: feasibility verdicts, approximate quantities/costs/emissions, and explanation traces.

6. Scope and Phasing
Phase 1 (MVP): Structural feasibility + greedy allocation + diagnostics
- Reachability checks (per region/year/timeslice/commodity demand)
- Cheapest-path pathway identification (top-N pathways)
- Greedy allocation with existing capacities/bounds
- Capacity expansion heuristic (add marginal capacity to relieve bottlenecks)
- Outputs: unmet demand, bottlenecks, pathway breakdown, cost/emissions summaries

Phase 2: Key policy constraints via shadow prices
- Emissions cap (global or by region) enforced via iterated carbon price
- Renewable share constraint via iterated penalty on non-renewable generation/energy
- Phase-out / minimum build constraints (hard bounds) as enforceable checks

Phase 3: Intertemporal and storage approximations
- Stock turnover approximation for stock-based sectors (transport/buildings/industry equipment)
- Basic storage shifting heuristic by timeslice price differentials (electricity, hydrogen, etc.)
- Simple build-rate / growth constraints

7. Functional Requirements
FR1. IR Adapter Layer
- Provide an adapter mapping from existing IR schema to internal pre-solver structures:
  - Set keys: region, year, timeslice, commodity/service, process/technology
  - Process coefficients: inputs/outputs (including efficiencies), joint products
  - Bounds: activity bounds, capacity bounds, resource supply bounds, trade bounds
  - Costs: variable OPEX, fixed OPEX, capex annualization factors, discounting inputs (if available)
  - Emissions: per-flow or per-activity emissions factors; mapping to emissions commodities/categories
  - Policy constraints: emissions caps, shares, min/max builds, bans/phase-outs

FR2. Reachability / Connectivity Diagnostics
- For each demand node (commodity/service, region, year, timeslice):
  - Determine if any supply pathway exists from any resource/source node.
  - If not reachable: report minimal “missing link” explanation:
    - missing producer edge, missing trade link, missing conversion, missing base commodity, etc.
- Must run fast and be available as a standalone compiler check.

FR3. Cheapest-Path Pathway Enumeration
- Build a directed graph for each (region, year, timeslice) or a factorized graph with timeslice modifiers.
- Edge cost is an “effective generalized marginal cost”:
  - gen_cost = var_cost + (annualized_capex / (AF * hours_in_slice)) + policy penalties (optional)
- Compute:
  - single cheapest pathway (Dijkstra) AND top-N cheapest pathways (Yen’s algorithm or k-shortest paths approximation)
- Output pathway sequences and cost/emissions decomposition.

FR4. Greedy Allocation Engine
- Allocate demand along cheapest pathways subject to bounds:
  - resource supply limits
  - capacity limits (existing + built)
  - activity bounds, availability factors
  - trade/transmission constraints if represented in IR
- If a pathway saturates: fallback to next-cheapest pathway.
- If all pathways saturate: trigger capacity expansion (FR5) or declare unmet demand.

FR5. Capacity Expansion Heuristic
- When unmet demand exists due to capacity bottlenecks:
  - Identify the marginal technology on the selected pathway responsible for the bottleneck.
  - Build Δcap sized to cover unmet output:
    - Δcap = unmet_output / (AF * hours_in_slice) (or equivalent, consistent with IR units)
  - Respect build limits if present; if violated → emit diagnostic.
- Update internal capacity state; re-run allocation for affected nodes.
- Maintain trace: why built, size, costs, and which constraint was binding.

FR6. Policy Constraint Enforcement (Phase 2+)
FR6.1 Emissions Cap via Shadow-Price Iteration
- If an emissions cap exists (global or subset):
  - Iterate a carbon price λ:
    - generalized cost = $cost + λ * tCO2e
  - Run allocation (FR4/FR5), compute total emissions, update λ using bisection/secant.
  - Stop when emissions within tolerance or λ exceeds max.
- If infeasible at very high λ: report “cap infeasible given technology set/bounds” with drivers.

FR6.2 Renewable (or other) Share via Penalty Iteration
- For a share constraint (e.g., renewables >= X%):
  - Apply penalty μ to “non-renewable” energy and iterate μ until share achieved or infeasible.
- Must identify the classification mapping (renewable vs non-renewable) from IR metadata/tags.

FR6.3 Hard Bounds: Bans/Phase-outs/Min Builds
- Translate these into hard availability/bounds for the heuristic engine.
- If violated: diagnostic with implicated techs/years/regions.

FR7. Outputs and Artifacts
The pre-solver must output both machine-readable and human-readable artifacts:
- presolve_summary.json (or equivalent):
  - feasibility verdicts
  - unmet demand table
  - emissions/cost summaries
  - implied carbon price (if emissions cap mode)
  - key pathway selections
- presolve_flows.parquet/csv (optional):
  - activity by process, output by commodity, capacity additions
- presolve_diagnostics.json:
  - typed diagnostics (error/warn/info), codes, locations in source (if available), IR references
- presolve_trace.jsonl:
  - decision trace events (path chosen, constraint binding, fallback, build action)
- presolve_report.md/txt:
  - concise human-readable report for CLI/CI logs

FR8. CLI / API Integration
- Provide a compiler flag:
  - vedalang compile --presolve (default on in dev mode; configurable in CI)
- Provide modes:
  - --presolve=off | basic | policy | full (phased feature gates)
- Provide output directory control:
  - --presolve-out=...
- Provide exit codes for CI:
  - non-zero on errors (unreachable demand, infeasible hard constraints)
  - warnings do not fail unless configured

8. Non-Functional Requirements
NFR1. Performance
- MVP target: 1e5–1e6 “edge operations” in < 2 seconds on a laptop.
- Must support incremental compilation: reuse graph structures when only small IR diffs occur (nice-to-have).

NFR2. Determinism and Reproducibility
- Stable ordering for ties (e.g., lexicographic tech ID).
- Record pre-solver version + config hash in outputs.

NFR3. Explainability
- Every demand allocation and capacity build must be traceable:
  - chosen pathway ID
  - marginal generalized cost components
  - binding constraint and slack values
  - alternative considered when fallback occurs

NFR4. Robustness
- Graceful handling of missing optional fields; emit warnings rather than crashing.
- Unit consistency checks with clear error messages where possible.

9. Technical Design
9.1 Internal Data Structures
- Node key: (commodity/service, region, year, timeslice)
- Process edge: maps input nodes → output nodes with coefficients and constraints.
- Capacity state: per (tech, region, year) capacity available; optionally vintaged.
- Resource state: per (resource, region, year, timeslice) remaining supply.
- Policy state: emissions budget, share targets, penalties (λ, μ).

9.2 Graph Construction
- Build adjacency lists from IR process definitions.
- Optional: build per-year graphs; treat timeslices as modifiers (AF, hours, demand scaling).
- Store:
  - min-edge-cost for Dijkstra
  - emissions intensity per edge
  - capacity requirement indicator (edge uses capacity vs pure trade/resource)

9.3 Algorithms
A. Reachability: BFS/DFS on unweighted graph ignoring costs (fast fail-fast).
B. Cheapest path: Dijkstra using generalized cost.
C. Top-N paths (optional): k-shortest paths with pruning; or compute 1 path plus a small set of near alternatives.
D. Greedy allocation:
   - iterate demands in descending size (or priority order)
   - allocate along cheapest path; update states
   - if blocked, use next path; if none, expand capacity if allowed
E. Shadow price iteration:
   - outer loop on λ (carbon) / μ (share)
   - inner run is allocation; compute residual; update penalty; converge

9.4 Storage Heuristic (Phase 3)
- Derive timeslice marginal prices from supply generalized costs.
- If price_spread > threshold/η:
  - shift energy via storage subject to power/energy bounds and η_roundtrip
- Iterate 1–2 times; record storage use and remaining unmet peak demand.

9.5 Stock Turnover (Phase 3)
- Represent stock-based service demands with simple cohorts:
  - stock[t] = stock[t-1]*(1-retire) + new_build[t]
- Serve service demand from stock with utilization limits.
- New-build choice via generalized cost ranking with short look-ahead (optional).

10. Diagnostics Specification
10.1 Diagnostic Levels
- ERROR: compile should fail (unreachable demand, hard infeasibility, unit/efficiency missing where required)
- WARN: suspicious but possibly intended (very high implied carbon price, extreme cost spikes, near-zero AF)
- INFO: useful context (dominant pathways, binding constraints)

10.2 Diagnostic Codes (initial list)
- E1001 UnreachableDemand
- E1002 NoProducerForCommodity
- E1003 NoTradeLink
- E1004 CapacityBoundPreventsFeasibility
- E1005 BuildLimitPreventsFeasibility
- E1006 EmissionsCapInfeasible
- W2001 ImpliedCarbonPriceHigh
- W2002 DemandMetOnlyViaExpensiveBackstop
- W2003 PossibleUnitMismatch
- W2004 NearZeroEfficiencyOrAF
- I3001 DominantPathway
- I3002 BindingBottleneck

Each diagnostic must include:
- code, level, message
- IR references (IDs of processes/commodities/constraints)
- scope (region/year/timeslice)
- suggestion (actionable remediation guidance)

11. Acceptance Criteria
AC1. For any model where demands are structurally reachable and bounds allow feasibility, pre-solver produces:
- zero unmet demand (within tolerance)
- a consistent set of flows/capacities that respects supported constraints
AC2. For any model with unreachable demand, pre-solver fails with E1001 and a minimal missing-link trace.
AC3. When emissions cap is enabled, pre-solver reports implied carbon price and meets the cap within tolerance, OR fails with E1006 and explanation.
AC4. Runtime meets performance targets on representative internal models.
AC5. Outputs are stable across runs and include traceability metadata.

12. Testing Strategy
- Unit tests:
  - graph construction from IR adapter mappings
  - reachability diagnostics
  - Dijkstra path correctness on toy graphs
  - greedy allocation state updates
  - capacity expansion sizing
  - shadow price convergence on small examples
- Regression tests:
  - curated suite of small-to-medium models with known qualitative outcomes
  - “golden” diagnostics for common failure modes
- Property-based tests (nice-to-have):
  - invariants: non-negative flows, respecting bounds, conservation constraints (within heuristic semantics)

13. Metrics / Telemetry (developer-facing)
- presolve_runtime_ms, edges, nodes, demands
- unmet_demand_total, #unreachable_demands
- #capacity_build_actions, total_capex_added
- emissions_total, implied_carbon_price (if enabled)
- #diagnostics by level

14. Implementation Plan (Suggested)
Milestone M1 (1–2 weeks): MVP reachability + diagnostics + report artifacts
Milestone M2 (2–4 weeks): cheapest-path + greedy allocation + capacity expansion; JSON/Parquet outputs
Milestone M3 (2–3 weeks): emissions cap via shadow price; CI integration gates
Milestone M4 (3–5 weeks): storage + stock turnover; performance tuning; incremental compile support

15. Risks and Mitigations
R1. IR complexity and heterogeneity
- Mitigation: strict adapter interface; start with a “supported subset” and emit warnings for unsupported constructs.
R2. Misleading results interpreted as “forecast”
- Mitigation: explicit labeling in outputs; document approximation scope; require opt-in for publishing estimates.
R3. Performance on very large models
- Mitigation: precompute graphs; prune k-shortest paths; caching per (region,year,timeslice); incremental compile.
R4. Storage/temporal coupling inaccuracies
- Mitigation: keep Phase 3 heuristic conservative; emphasize diagnostics over precise quantities.

16. Open Questions for Dev Planning (to resolve during design)
- Which IR fields/tags define “renewable” classification and emissions categories?
- How are timeslices represented (hours/weights) in IR, and do we have consistent AF definitions?
- Do we need to support multi-output processes and byproducts in MVP, or only single-output pathways?
- What is the minimal supported set of policy constraints for Phase 2 in your current roadmap?
- Do we want a fallback “tiny LP min-cost flow” mode for robustness (still pre-TIMES), and do we have an internal LP solver dependency policy?
