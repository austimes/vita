PRD: Emissions-as-Attributes with Human-Readable Namespaces (VedaLang)

Summary
Adopt a namespace + type scheme in VedaLang that keeps the language obvious and human-readable, while compiling deterministically to TIMES / VEDA core commodity sets and process groups.

Key principle:
- Emissions are attributes recorded in an emissions ledger, not “useful outputs”.
- Users write emission:co2, primary:natural_gas, service:space_heat, material:co2 (etc.).
- The compiler maps these to VEDA commodity Csets (NRG/MAT/DEM/ENV/FIN) and process group codes (ELE/CHP/PRE/…).

This reduces semantic ambiguity, improves RES diagnostics, and makes export/mapping to TIMES more reliable.


Motivation / Problem
Current models can represent emissions two ways at once:
- as regular outputs (e.g., co2), and/or
- as emission coefficients/structures (e.g., emission_factors).

Downstream, this confuses:
- RES diagnostics (multi-output classification, output accounting),
- exporters/mappers expecting emissions via emission constructs,
- any “production by commodity” logic (CO2 shows up like a co-product).

We want a modelling convention that is:
1) unambiguous, 2) lintable, 3) friendly to humans, 4) maps cleanly to TIMES/VEDA.


Goals
1. Obviousness-first syntax: namespaces should read like English.
2. Strict semantic separation:
   - ledger emissions ≠ physical CO2 stream
3. Deterministic compilation to VEDA/TIMES core sets:
   - Commodity types (Csets): NRG/MAT/DEM/ENV/FIN
   - Process groups (e.g., ELE/CHP/PRE/…): support compile-time mapping
4. Support negative emissions (DAC/LULUCF) in a physical, explainable manner.


Non-goals
- Not redesigning all process semantics or adding a full CCS subsystem spec.
- Not enforcing a specific unit system beyond existing unit/type checks (though this makes it easier).


Background: TIMES/VEDA “Core” Sets We Must Map To

Commodity types (Csets) in VEDA (~FI_COMM)
VEDA’s commodity table supports Csets values:
- NRG (energy), MAT (material), DEM (demand service), ENV (emissions), FIN (financial).

Process groups
Canonical process group codes include:
- CHP, DISTR, DMD, ELE, HPL, MISC, PRE, REF, RENEW (and others depending on tooling).


Proposed User-Facing Namespaces (Obvious Names)

Commodity namespaces (user-facing → VEDA Csets)

- primary:   Meaning: primary combustible/extractable fuels               Compiles to: NRG
- secondary: Meaning: secondary carriers (electricity, hydrogen, etc.)    Compiles to: NRG
- material:   Meaning: physical materials/streams (including captured CO2)    Compiles to: MAT
- service:    Meaning: demand/service commodities (heat service, pkm, etc.)   Compiles to: DEM
- emission:   Meaning: environmental ledger species (CO2, CH4, N2O)           Compiles to: ENV
- money:      Meaning: monetary/financial commodities if used                 Compiles to: FIN

Design rule: the 3-letter TIMES codes never need to appear in user-authored VedaLang. They are compiler output.


Emissions-as-Attributes: Core Semantics

Principle
- emission:* commodities are not tradable flows. They are ledger entries tied to process activity.
- Therefore, they must not appear in inputs/outputs.
- Emissions enter the model only via emissions metadata such as emission_factors.

Example: Gas heater (toy_buildings fix)

process_variants:
  - id: gas_heater
    role: provide_space_heat
    inputs:
      - commodity: primary:natural_gas
    outputs:
      - commodity: service:space_heat
    emission_factors:
      emission:co2: 0.056

Compile intent:
- primary:natural_gas → NRG commodity
- service:space_heat → DEM commodity
- emission:co2 → ENV emission commodity


DAC / LULUCF: “Consuming emissions” without breaking physicality

Key point
DAC and LULUCF should not “consume emission:co2” as an input flow because emission:* is not a flow network. Instead they produce a negative ledger entry (i.e., net removal).

So:
- Removals = negative emission_factors on emission:co2.
- If you also need explicit CO2 transport/storage, that’s a material stream: material:co2.

DAC pattern (minimal, removal-only)

process_variants:
  - id: dac
    role: remove_co2
    inputs:
      - commodity: secondary:electricity
    emission_factors:
      emission:co2: -1.0

DAC with explicit captured stream (for CCS infrastructure)

process_variants:
  - id: dac
    role: remove_co2
    inputs:
      - commodity: secondary:electricity
    outputs:
      - commodity: material:co2
    emission_factors:
      emission:co2: -1.0

Interpretation:
- emission:co2: -1.0 = removes from ENV ledger.
- material:co2 = captured CO2 stream that can be compressed/transported/stored.

LULUCF sequestration pattern

process_variants:
  - id: afforestation
    role: sequester_carbon
    inputs:
      - commodity: service:land
    emission_factors:
      emission:co2: -0.5

This stays “physical” in the sense that you’re modelling a sequestration activity that reduces atmospheric CO2, without turning the emissions ledger into a tradable commodity.


Lint Rules (Must/Should)

L1 (Error): emission:* MUST NOT appear in inputs or outputs
Rationale: prevents emission-as-coproduct ambiguity.

L2 (Error): emission_factors keys MUST be emission:*
Rationale: prevents confusing a physical commodity with an emissions ledger species.

L3 (Allowed + Warn-on-missing-doc): negative emission factors allowed
- Permit emission_factors[emission:co2] < 0 for removals.
- Recommend a doc field when negative (e.g., notes:) to improve auditability.

L4 (Error): mixed-mode emissions representation
- If the same logical species is represented both as an emission ledger entry and as a flow (e.g., outputs: [emission:co2] plus emission_factors), error (covered by L1 but keep for clearer messaging).

L5 (Migration aid): un-namespaced co2 is ambiguous
- Warn: “Did you mean emission:co2 or material:co2?”
- Offer auto-fix suggestions.


Process Group Friendly Aliases (Optional but recommended)
Allow intuitive process tags in VedaLang that compile to TIMES process groups (PRC_GRP style codes) such as ELE, CHP, PRE, etc.

Examples (user-facing → compiled code):
- power_plant → ELE
- combined_heat_power → CHP
- generic_process → PRE
- demand_device → DMD
- distribution → DISTR
- refinery → REF
- renewable → RENEW

(Exact alias list can be minimal; the compiler mapping table is the contract.)


Compiler Responsibilities
1. Parse namespaces and map commodities to VEDA Csets (NRG/MAT/DEM/ENV/FIN).
2. Ensure emission:* never participates in flow balance equations; it only affects:
   - emission caps/taxes/reporting
   - objective terms if configured
3. Export:
   - emission:* → TIMES emission constructs
   - energy/material/service → normal process in/out coefficients
4. If TIMES export path cannot represent negative emission coefficients directly in a chosen table, the exporter may implement an equivalent transformation — but VedaLang semantics remain “negative ledger entry”.


Implementation Plan

Phase 1 — Namespaces + Lint
- Add namespace parsing + canonical printing.
- Implement L1–L5.
- Update toy models (toy_buildings) to remove emission commodities from outputs.

Phase 2 — Export mapping
- Route emission:* strictly through emission constructs.
- Preserve material:co2 as normal MAT flow for CCS chains.

Phase 3 — Docs + “VedaLang skill.md”
- Document namespace conventions and the “ledger vs flow” split.
- Provide short cookbook examples: combustion, DAC, LULUCF, CCS chain.


Acceptance Criteria
- vedalang lint --llm-assess on toy_buildings yields no warning about emissions-as-outputs.
- Linter errors on any inputs/outputs containing emission:*.
- DAC/LULUCF examples validate with negative emission factors.
- Export preserves deterministic mapping to VEDA Csets and accepted process group codes.
