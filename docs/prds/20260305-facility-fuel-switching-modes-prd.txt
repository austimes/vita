PRD: Facility Fuel-Switching via Modes (Physical-Only, LP-Safe)
==============================================================

1. Purpose
----------

Enable VedaLang to represent fuel switching (and “upgrades”) for facilities in a way that is:

- Explainable to stakeholders (industry + policy)
- Semantically clean in VedaLang (upgrades ≠ new technologies)
- Physical-only RES compliant (no fictitious “capability” commodities)
- LP-only (no mixed-integer required for core functionality)

Supports:
- Capacity-based upgrade costs (retrofit CAPEX)
- Gradual switching (ramp limits)
- No backsliding (irreversible conversion away from dirtier fuels)


2. Problem Statement
--------------------

We need to prevent unrealistic behaviour where an LP can “switch” a facility from coal → gas → coal across
periods as emissions pressure changes. We also need to represent retrofit costs as capacity-based investment
decisions, while letting the model choose when and how much to retrofit.

A naïve single-process multi-fuel representation lacks an explicit decision variable to trigger retrofit CAPEX
and to enforce irreversibility. A common approach is to introduce a “capability commodity”; however, this
violates the project’s physical-only RES principle.


3. Proposed Solution: “Modes” as Operational Configurations
-----------------------------------------------------------

Introduce “mode” as a first-class concept distinct from “variant”.

- Variants remain for technology archetypes (e.g., kiln A vs kiln B, tech A vs tech B).
- Modes represent how the same facility operates under different fuels/configurations
  (coal, NG retrofit, H₂ retrofit, etc.).

Key idea:
For each facility-variant, compile to one process per mode that:
- consumes a single physical fuel commodity (coal, NG, H₂, etc.)
- produces the same physical service commodity (e.g., useful high-temp heat)
- has its own capacity decision variable and (optional) CAPEX

Retrofit costs are implemented as CAPEX on non-baseline mode capacity.
No fictitious commodities are introduced.


4. VedaLang Model Semantics
---------------------------

4.1 Entities

Facility
- An asset with a fixed baseline capacity (cap_base) of a service (e.g., MWth useful heat).
- A set of variants; each variant has a set of modes.

Variant
- Technology archetype for the facility role.

Mode
- Operational configuration (typically fuel-specific).
- Can be baseline (existing) or retrofit-enabled (investable).

4.2 Minimal Authoring Schema (illustrative)

facility:
  id: alumina_calciner_1
  role: high_temp_heat
  cap_base:
    value: 500
    unit: MWth
  baseline_mode: coal

  variants:
    - id: calciner_standard
      service_output: NRG:useful_heat_high_temp
      modes:
        - id: coal
          fuel_in: NRG:coal
          capex: 0
          existing: true
        - id: retrofit_to_ng
          fuel_in: NRG:natural_gas
          capex: 1200   # $/kWth (retrofit-only)
          existing: false
        - id: retrofit_to_h2
          fuel_in: NRG:hydrogen
          capex: 2500
          existing: false


5. Compilation Design (TIMES/VEDA)
----------------------------------

5.1 Naming Standard (compiler output)

Canonical process name pattern:

  FAC::{facility_id}::ROLE::{role_id}::VAR::{variant_id}::MODE::{mode_id}

Example:
- FAC::alumina_calciner_1::ROLE::high_temp_heat::VAR::calciner_standard::MODE::coal
- FAC::alumina_calciner_1::ROLE::high_temp_heat::VAR::calciner_standard::MODE::retrofit_to_ng
- FAC::alumina_calciner_1::ROLE::high_temp_heat::VAR::calciner_standard::MODE::retrofit_to_h2

(Short-form aliases are fine if token order is preserved and reversible.)

5.2 Process I/O Structure (physical-only)

For each mode process:
- Inputs: exactly one fuel commodity (physical)
- Outputs: common service commodity (physical/service, e.g., useful heat)

No “capability” commodity.

5.3 Capacity representation

Baseline mode (existing: true):
- Capacity is fixed (or upper-bounded then constrained) to represent existing plant stock.
- CAPEX = 0 (unless you want fixed O&M etc.)

Retrofit modes (existing: false):
- Capacity is investable (decision variable).
- CAPEX = retrofit cost only (do NOT include the full “new build” plant CAPEX).

5.4 “Single physical plant” coupling constraint

To prevent double-counting capacity across modes:

For each facility-variant and period t:

  CAP_coal,t + CAP_ng,t + CAP_h2,t ≤ CAP_base

Optionally enforce equality if you want all capacity allocated:

  CAP_coal,t + CAP_ng,t + CAP_h2,t = CAP_base

Recommendation:
Use “≤” as the default to avoid infeasibility when some modes are disallowed in certain periods or when the
facility can be partially idle. “=” can be enabled as a stricter option when appropriate.


6. No-Backsliding Specification
-------------------------------

6.1 Meaning

Once capacity has been converted to a “cleaner” mode, it cannot revert to a dirtier mode later.

Implemented as monotonic non-decrease of retrofit-mode capacities (or shares).

6.2 Mode ladder ordering

Require an explicit ordering of modes by conversion level (or infer ordering from a policy ladder list).

Example ladder:
- coal (0)
- retrofit_to_ng (1)
- retrofit_to_h2 (2)

6.3 LP constraints (capacity-based irreversibility)

For each retrofit mode m:

  CAP_m,t ≥ CAP_m,t-1

To ensure coal cannot return (tighten coal capacity ceiling as retrofit grows):

  CAP_coal,t ≤ CAP_base − Σ_{m ∈ retrofit_modes} CAP_m,t

This is sufficient to prevent coal expansion once any retrofit capacity is built.

Optional (strict irreversibility on coal operation):
If the intent is “converted capacity can never burn coal again”, use the coupling + monotonicity above and
ensure coal’s feasible capacity shrinks exactly with retrofit capacity. (This is already achieved when coal is
bounded by CAP_base minus retrofit capacities.)


7. Switching Speed / Realism
----------------------------

7.1 Ramp constraints (capacity conversion rate)

For each retrofit mode m:

  CAP_m,t − CAP_m,t-1 ≤ r_m × CAP_base

Where r_m is a scenario/assumption lever (e.g., 0.25/yr means 4 years to fully convert).

7.2 Defaults (as scenario-layer assumptions)

Provide global defaults by transition type (overridable), e.g.:
- coal→gas: r = 0.25–0.50 per year
- gas→H₂: r = 0.10–0.33 per year

Exact values are assumptions and should live in the scenario/assumption layer, not hard-coded in this spec.


8. Cost Accounting Requirements
-------------------------------

8.1 Retrofit CAPEX

Retrofit mode CAPEX is charged on capacity additions to that mode (standard TIMES investment logic).

This matches “capacity-based upgrade cost” without needing MIP.

8.2 Prevent “pay but never use” (optional)

LP may build retrofit capacity and then not dispatch it if later economics shift.

Mitigations (optional, selectable):
- Add a small fixed O&M on retrofit mode capacity (discourages unused build).
- Add minimum utilisation constraints (more intrusive).
- Add scenario rules: if a mode is built, it must meet ≥X% of service in subsequent periods (intrusive).

Default:
Do nothing unless this artefact becomes material in results review.


9. Implementation Plan
----------------------

9.1 Compiler changes

1) Add “mode” to the internal facility AST.
2) For each facility × variant × mode, emit a TIMES process:
   - Input fuel commodity
   - Output service commodity
   - Efficiency and emissions attributes as usual
3) Identify baseline mode and set its capacity handling:
   - Fixed to cap_base, OR bounded + constrained.
4) For retrofit modes:
   - Create investable capacity with CAPEX = retrofit cost only.
5) Emit coupling constraint across mode capacities for each facility-variant.
6) If no_backsliding: true, emit monotonic constraints for retrofit modes.
7) If ramp_rate provided, emit ramp constraints.

9.2 Where to encode constraints in TIMES

Use TIMES user constraints (UC/UC_T) or equivalent:

- Σ CAP(mode) ≤ CAP_BASE
- CAP(mode,t) − CAP(mode,t-1) ≤ ramp
- CAP(mode,t) − CAP(mode,t-1) ≥ 0   (monotone)

Keep generated constraints in a dedicated output file per facility-group for traceability.

9.3 Traceability & explainability

Emit a generated artifact per facility:
- facility_<id>__modes.md (human readable)
- Lists emitted processes and constraints in plain English
- Provides the “industry story”:
  - “Mode capacity represents portion of plant configured for that fuel.”
  - “Retrofit CAPEX is paid when mode capacity is added.”
  - “No-backsliding enforced by monotone constraints.”


10. Worked Example
------------------

Goal:
Show how the LP chooses WHEN to retrofit, pays only retrofit CAPEX, and cannot backslide.

Setup:
- Periods: 2025, 2030, 2035 (5-year steps for illustration)
- Facility: alumina_calciner_1
- Baseline capacity: CAP_base = 500 MWth (useful heat capacity)
- Modes:
  - coal (existing, CAPEX=0)
  - retrofit_to_ng (CAPEX = 1200 $/kWth)
  - retrofit_to_h2 (CAPEX = 2500 $/kWth)
- No-backsliding enabled for retrofit modes (NG and H2 are monotone)
- Ramp limits:
  - NG ramp: r_ng = 0.50 per period (i.e., ≤ 250 MWth per period)
  - H2 ramp: r_h2 = 0.20 per period (i.e., ≤ 100 MWth per period)

(1) Emitted processes (conceptual names)

P_COAL = FAC::alumina_calciner_1::ROLE::high_temp_heat::VAR::calciner_standard::MODE::coal
P_NG   = FAC::alumina_calciner_1::ROLE::high_temp_heat::VAR::calciner_standard::MODE::retrofit_to_ng
P_H2   = FAC::alumina_calciner_1::ROLE::high_temp_heat::VAR::calciner_standard::MODE::retrofit_to_h2

All three produce the same output commodity:
  NRG:useful_heat_high_temp

Inputs differ:
  P_COAL consumes NRG:coal
  P_NG consumes   NRG:natural_gas
  P_H2 consumes   NRG:hydrogen

(2) Core constraints

Capacity coupling (single physical plant), per period t:
  CAP_COAL,t + CAP_NG,t + CAP_H2,t ≤ 500

No-backsliding (monotone), per period transition:
  CAP_NG,2030 ≥ CAP_NG,2025
  CAP_NG,2035 ≥ CAP_NG,2030
  CAP_H2,2030 ≥ CAP_H2,2025
  CAP_H2,2035 ≥ CAP_H2,2030

Coal ceiling tightens as retrofit grows, per period t:
  CAP_COAL,t ≤ 500 − (CAP_NG,t + CAP_H2,t)

Ramp limits, per transition:
  CAP_NG,2030 − CAP_NG,2025 ≤ 250
  CAP_NG,2035 − CAP_NG,2030 ≤ 250
  CAP_H2,2030 − CAP_H2,2025 ≤ 100
  CAP_H2,2035 − CAP_H2,2030 ≤ 100

(3) Illustrative optimal path (one possible outcome)

Assume:
- Between 2025 and 2030, emissions pressure increases and NG becomes attractive.
- Between 2030 and 2035, hydrogen becomes attractive.

The LP may choose:

2025:
  CAP_NG,2025 = 0
  CAP_H2,2025 = 0
  CAP_COAL,2025 = 500

2030 (NG retrofit constrained by ramp to ≤250):
  CAP_NG,2030 = 250
  CAP_H2,2030 = 0
  CAP_COAL,2030 = 250   (since coal ceiling tightens)

2035 (H2 ramp to ≤100 per period, plus could add more NG):
  CAP_NG,2035 = 350   (adds 100 MWth NG within ramp)
  CAP_H2,2035 = 100   (adds 100 MWth H2 within ramp)
  CAP_COAL,2035 = 50  (coal shrinks accordingly)

(4) Retrofit CAPEX charged (capacity-based)

Retrofit costs are paid on capacity additions to retrofit modes:

NG additions:
- 2025→2030: ΔCAP_NG = 250 MWth
- 2030→2035: ΔCAP_NG = 100 MWth

H2 additions:
- 2025→2030: ΔCAP_H2 = 0
- 2030→2035: ΔCAP_H2 = 100 MWth

Convert MWth to kWth for the $/kWth CAPEX:
- 250 MWth = 250,000 kWth
- 100 MWth = 100,000 kWth

CAPEX incurred (undiscounted, before annualisation):
- NG: (250,000 + 100,000) × 1200 = 350,000 × 1200 = 420,000,000 $
- H2: 100,000 × 2500 = 250,000,000 $

Total retrofit CAPEX = 670,000,000 $ (then annualised via TIMES finance).

(5) No backsliding demonstrated

Suppose in 2035 coal becomes “cheap” again or emissions pressure relaxes.

The LP cannot backslide because:
- CAP_NG and CAP_H2 cannot decrease (monotone constraints).
- CAP_COAL is bounded by 500 − (CAP_NG + CAP_H2).

So coal capacity cannot re-expand above 50 MWth in 2035 given the chosen retrofit path.

Dispatch can still choose coal within the remaining coal capacity (if allowed), but the system cannot revert to
a larger coal share without violating constraints.


11. Acceptance Criteria
-----------------------

- A facility can progressively shift coal → gas → H₂ across periods.
- Model cannot increase coal mode capacity after building retrofit capacity (no backsliding).
- Retrofit CAPEX is paid when retrofit mode capacity is added, not per PJ of fuel use.
- No fictitious “capability” commodities appear in the RES.
- All formulation remains linear (LP).
- Generated process names are stable and fully qualified by facility/role/variant/mode.


12. Non-Goals (for this iteration)
----------------------------------

- Discrete “fixed-charge” retrofit projects (requires MIP)
- Unit-level indivisibilities (e.g., exactly 3 trains)
- Detailed construction lead times (can be added later via lagged availability)


13. Notes / Design Rationale
----------------------------

This approach treats “retrofit” as building mode capacity rather than introducing an explicit “upgrade kit
commodity”. The coupling constraint ensures mode capacity is a partition of a single physical plant, so the
model is not “building a new plant”; it is reallocating the existing plant’s capacity across allowable
operating configurations, paying only retrofit CAPEX for converted shares.
