PRD v2.3
Typed DSL for VEDA-TIMES Identity, Naming, and Classification

Status: Draft for implementation
Audience: Model tooling developers, AusTIMES core modellers
Scope: Naming, identity, tagging, sets, and registry generation

=====================================================================
1. PURPOSE
=====================================================================

Define a robust, typed identity system for processes and commodities that:

- Eliminates ad-hoc string naming
- Enforces semantic consistency via compiler rules
- Supports multi-region modelling explicitly
- Enables modern tooling (LLMs, Sankey diagrams, dashboards)
- Keeps VEDA-TIMES inputs minimal and free of cruft

=====================================================================
2. CORE ARCHITECTURAL DECISIONS
=====================================================================

1) Geography handling
- Geography is a first-class index in the model.
- Regions/subregions are explicit fields in tooling and emitted tables.

2) Commodity IDs
- Tradable, service, and emission commodities DO NOT embed geography in IDs.
- Geography is handled by indexing, not naming.

3) Process IDs
- Processes DO embed geography in IDs (Option B).
- Geography is also an explicit indexed field.
- Compiler enforces consistency between the two.

Summary:

Entity                  Geo in ID   Geo indexed
------------------------------------------------
Tradable commodity      No          Yes
Service commodity       No          Yes
Emission commodity      No          Yes
Process                 Yes         Yes

=====================================================================
3. COMMODITY KINDS
=====================================================================

Three mutually exclusive kinds:

1) TRADABLE
   - Conserved physical flows.
   - Can be produced, consumed, transported, stored, traded.
   - No demand context.
   - Examples: electricity, coal, gas, hydrogen, limestone, biomass.

2) SERVICE
   - Contextual demand satisfaction.
   - Not traded or transported.
   - Defined per sector/segment.
   - Examples: lighting (commercial office), space heating (residential).

3) EMISSION
   - Byproduct / waste flows.
   - Attributed by region index.
   - Aggregated in post-processing unless policy requires otherwise.
   - Examples: CO2, CH4, N2O.

=====================================================================
4. IDENTIFIER GRAMMAR
=====================================================================

General rules:
- Uppercase emitted IDs
- Allowed characters: A-Z 0-9 : . _
- ':' separates fields
- '.' expresses hierarchy
- '_' separates words inside tokens

------------------------------------------------
4.1 Tradable commodities
------------------------------------------------

Form:
C:{COMMODITY}

Examples:
C:ELC
C:COA
C:H2
C:LMS

------------------------------------------------
4.2 Service commodities
------------------------------------------------

Form:
S:{SERVICE}:{SECTOR}.{SEGMENT}[.{SUBSEGMENT}]

Examples:
S:LGT:COM.OFF
S:LGT:COM.OFF.A_GRADE
S:SPH:RES.DET

------------------------------------------------
4.3 Emission commodities
------------------------------------------------

Form:
E:{GAS}

Examples:
E:CO2
E:CH4

------------------------------------------------
4.4 Process identifiers (Option B)
------------------------------------------------

Form:
P:{TECH}:{ROLE}:{GEO}[:{SECTOR}.{SEGMENT}[.{SUBSEGMENT}]][:{VARIANT}][:{VINTAGE}]

Examples:
P:STC:GEN:NSW.NCC
P:LED:EUS:NSW.NCC:COM.OFF
P:RCD:CNV:QLD.BNE
P:PCC:CAP:NSW.NCC:CCS90

Rules:
- Inputs/outputs NEVER encoded in IDs
- Context required only for end-use/service roles
- Compiler lints if geo in ID mismatches indexed geo

=====================================================================
6. TAG TAXONOMY
=====================================================================

Tags exist in two classes:

------------------------------------------------
6.1 Class A - Structural (emitted to VEDA)
------------------------------------------------

Used for constraints and scenario logic:

- technology
- role
- sector / segment / subsegment (where relevant)
- variant (if constrained)
- vintage (if constrained)

------------------------------------------------
6.2 Class B - Descriptive (registry-only by default)
------------------------------------------------

Used for analytics and visualisation:

- sankey_stage
- display_label
- source_reference
- facility_id
- taxonomy mappings (IPCC, ANZSIC)

Compiler default:
- Emit Class A only to VEDA
- Emit ALL tags to registry.json
- Descriptive tags opt-in if needed

=====================================================================
7. SANKEY SUPPORT
=====================================================================

Tag: sankey_stage (Class B by default)

Purpose:
- Deterministic column assignment for Sankey diagrams

Allowed values:

SUP - primary supply / imports
PRC - processing / upgrading
XFR - transmission / transport
STO - storage
GEN - generation
END - end-use equipment
SRV - service delivery
EMI - emissions sink
CCS - capture / transport / storage
EXP - exports

Rules:
- Every process must have exactly one sankey_stage
- Compiler enforces consistency with ROLE

=====================================================================
8. SEGMENT HIERARCHY
=====================================================================

Structure:
sector.segment.subsegment

Rules:
- sector.segment required for SERVICE commodities
- subsegment optional
- Forbidden for TRADABLE and EMISSION commodities
- Required for ROLE=EUS processes

Examples:
COM.OFF
IND.METALS.ALUMINA

=====================================================================
9. DATA DICTIONARY
=====================================================================

------------------------------------------------
Process tags
------------------------------------------------

technology
- Meaning: canonical technology family
- Structural: Yes
- Example: STC, LED

role
- Meaning: functional position in system
- Structural: Yes
- Example: GEN, EUS, CNV
- Lint: EUS must output service commodity

sector.segment.subsegment
- Meaning: demand/institutional context
- Structural: Conditional
- Required when ROLE=EUS

variant
- Meaning: behavioural modifier
- Structural: Optional
- Example: CCS90

vintage
- Meaning: build cohort
- Structural: Optional
- Example: EXIST, NEW

sankey_stage
- Meaning: Sankey column
- Structural: No (default)
- Example: GEN

------------------------------------------------
Commodity tags
------------------------------------------------

kind
- Meaning: TRADABLE / SERVICE / EMISSION
- Structural: Registry only

context
- Meaning: sector.segment(.subsegment)
- Required for SERVICE only

=====================================================================
10. COMPILER OUTPUTS
=====================================================================

Must emit:

1) VEDA tables
   - processes
   - commodities
   - structural sets only

2) registry.json
   - full structured identity
   - full expansion of abbreviations
   - all descriptive tags

3) lint report
   - invalid combinations
   - missing required tags
   - geo mismatches

Optional:
- Sankey-ready edge list
- post-processing mapping tables

=====================================================================
11. ACCEPTANCE CRITERIA
=====================================================================

- No modeller writes raw VEDA IDs
- Abbreviations compiler-validated
- MIMO processes have short IDs
- Sankeys require no manual tuning
- Emissions computed in post-processing
- Geography refactors do not rename commodities

=====================================================================
12. NON-GOALS
=====================================================================

- Encoding accounting logic in naming
- Free-form tags affecting optimisation
- Regex-based selection as primary mechanism
