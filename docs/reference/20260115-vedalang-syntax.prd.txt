VedaLang RES Composition Syntax (Roles, Variants, Segments)
Product Requirements Document (PRD)

Version: 0.1 (Draft)
Date: 2026-01-15
Owner: (TBD)
Audience: VedaLang compiler/tooling developers
Status: Proposed (ready for implementation planning)

=====================================================================
1. Executive Summary
=====================================================================

We will make VedaLang RES authoring simpler and more composable by separating:

- Topology intent (what transformations exist): `roles`
- Technology implementation (how transformations are realized): `variants`
- Placement/availability (where implementations exist): `availability` over regions/segments

This PRD also standardizes VedaLang naming/semantics:

- Commodities are nouns (always).
- Roles are verb-phrases (always).
- Variants are technology-nouns/adjectives (e.g., led_lighting, coal_steam_turbine).

The compiler will deterministically lower these constructs to TIMES/VEDA entities:

- Commodity-type membership will align with TIMES hard-coded groups from `times/maplists.def`:
  COM_TYPE = { DEM, NRG, MAT, ENV, FIN }.
- Process-group membership will be compiler-owned (no user-authored "sets").

We explicitly do NOT provide an “escape hatch” for advanced/raw VEDA tags yet.

=====================================================================
2. Problem Statement
=====================================================================

Today, VedaLang RES structure is defined implicitly by process inputs/outputs and
user-authored names/fields such as `technology`, `sets`, and commodity `context`.
This creates confusion:

- Names embed semantics (e.g., `S:RSD:RES.ALL`), forcing readers to decode whether
  something is a process or commodity from the name alone.
- Multiple orthogonal classification knobs (commodity kind vs context; process
  role vs technology vs sets) make models hard to reason about and hard to lint.
- Composability is weak: assembling a RES from reusable components is cumbersome,
  and VEDA/TIMES “scenario override” workflows do not support rapid modular RES composition.

We want VedaLang to support modular RES authoring: define reusable topology
components once, then instantiate them across regions and sectoral segments.

=====================================================================
3. Goals / Non-Goals
=====================================================================

Goals
G1. A simple, opinionated syntax for RES structure that is easy to read and compose.
G2. Deterministic lowering to TIMES/VEDA structures with compiler-owned naming and sets.
G3. Support substitutable technology options cleanly: multiple variants can satisfy
    the same role (solver chooses a mix unless constrained).
G4. Explicit segmentation control: coarse sector-only models and fine end-use models
    should be a syntactic configuration change, not a naming refactor.
G5. Make it hard to create semantically ambiguous models; provide high-signal diagnostics.

Non-goals (for this PRD)
NG1. No raw VEDA tag/column “escape hatch” in the VedaLang surface syntax.
NG2. No discrete/integer “choose exactly one technology” semantics (TIMES is linear).
NG3. No new reporting taxonomy/labels beyond what is required for correctness.
NG4. No Mermaid parser as a required input format (a graph syntax may be added as sugar later).

=====================================================================
4. Terminology
=====================================================================

- Commodity: a noun representing a balanced flow node in the RES graph.
  Examples: electricity, natural_gas, lighting, space_heat, co2.

- Process role: an abstract transformation (no costs/efficiencies), used to define topology
  and to group substitutable technology options.
  Example: deliver_lighting (electricity -> lighting).

- Process variant: a concrete technology implementing one role. Variants carry parameters:
  efficiencies, costs, lifetimes, emissions coefficients, etc.
  Example: led_lighting_device implements deliver_lighting.

- Segment: a demand-side context. Initially scoped to (sector, end_use) with optional depth.
  Examples: RES, COM (coarse); RES.lighting, RES.heating (fine).

- Availability: where a variant (or role) exists. Availability is expressed over region and
  optionally segment.

=====================================================================
5. Design Principles
=====================================================================

P1. Separate “what exists” from “where it exists.”
    - Roles and commodities define the RES vocabulary.
    - Availability defines geography/segment placement.

P2. Compiler-owned determinism.
    - User does not specify TIMES/VEDA set membership or encoded IDs.
    - Compiler generates stable symbols and sets based on VedaLang semantics.

P3. Minimal semantics surface.
    - Commodity classification: one primary axis (kind → TIMES COM_TYPE).
    - Process classification: role → compiler-chosen PRC_GRP and required VEDA patterns.

P4. No silent structural auto-additions.
    - The compiler/linter may suggest missing prerequisites (“no supplier for coal”),
      but should not silently add mining/import processes by default.

=====================================================================
6. Functional Requirements (Syntax)
=====================================================================

6.1 Model dimensions
FR1. VedaLang must define regions and milestone years, as today.

6.2 Segmentation
FR2. VedaLang must provide an explicit, small segmentation vocabulary for demand-side modeling.
Initial scope:
  - `sector` is drawn from TIMES DEM_SECT in `times/maplists.def`: { RES, COM, IND, TRN, AGR, NE, OTH }.
  - Optional end-use dimension (“end_use”) to increase granularity within a sector.

Proposed representation (draft):

segments:
  # coarse
  sectors: [RES, COM]

  # optional fine granularity
  end_uses: [lighting, heating, cooling, hotwater]

Semantics:
  - If `end_uses` is absent: segment keys are just sectors (RES, COM).
  - If `end_uses` present: segment keys are sector.end_use (RES.lighting, RES.heating, ...).

6.3 Commodities
FR3. Commodities are declared once as nouns with stable user-facing IDs.
Required fields:
  - id: snake_case noun (e.g., electricity, lighting)
  - kind: carrier | service | material | emission
  - unit
Optional fields:
  - tradable: boolean (default true for carrier/material, default false for service/emission)

6.4 Process roles (topology)
FR4. Process roles define abstract transformations and are used for composition.
Required fields:
  - id: verb_phrase (e.g., deliver_lighting, generate_electricity)
  - inputs: list of commodity references (by id)
  - outputs: list of commodity references (by id)
Optional fields:
  - stage: supply | conversion | distribution | end_use | storage | trade | sink
    (stage is descriptive; used for sankey/layout only; does not affect optimization semantics.)

6.5 Process variants (technology)
FR5. Process variants implement exactly one role and provide numeric parameters.
Required fields:
  - id: technology identifier (snake_case)
  - role: reference to a process_role id
Optional (initial):
  - efficiency and/or input/output coefficients
  - lifetime
  - costs (capex, fom, vom)
  - emissions factors (mapping to emission commodities)
  - bounds defaults (optional; scenario and instance-specific bounds remain separate)

6.6 Availability (where variants exist)
FR6. Availability defines in which regions and segments variants are available.
Availability is required for any variant that should appear in the compiled model.

Proposed representation (draft):

availability:
  - variant: led_lighting_device
    regions: [SINGLE]
    sectors: [RES, COM]         # or segment keys (RES.lighting, ...)

  - variant: simple_generator
    regions: [SINGLE]
    # no sectors: applies globally in the region unless role requires segment

6.7 Demands (scenario parameters)
FR7. Demands are specified against service commodities (kind=service) and segments.
Demands are declared independently of which variants exist; feasibility is validated by the linter.

=====================================================================
7. Mapping to TIMES / VEDA (Lowering Requirements)
=====================================================================

7.1 Commodity group mapping (TIMES hard-coded)
FR8. The compiler must map VedaLang commodity kinds to TIMES COM_TYPE from `times/maplists.def`:
  - kind=service  → COM_TYPE = DEM
  - kind=carrier  → COM_TYPE = NRG
  - kind=material → COM_TYPE = MAT
  - kind=emission → COM_TYPE = ENV

Note: COM_TYPE also includes FIN in TIMES. We do not model FIN as a user-facing commodity kind;
FIN membership is compiler-owned and only emitted if/when required by TIMES patterns.

7.2 Sectoral scoping of non-tradable commodities (especially services)
FR9. If a commodity is non-tradable and is referenced in multiple sectors/segments,
the compiler must generate scoped TIMES commodity symbols per segment key.
User still declares the commodity once.

Example:
  VedaLang commodity: lighting (service)
  Segments used: RES, COM
  Compiler-generated internal symbols:
    lighting@RES, lighting@COM

7.3 Process group mapping (compiler-owned)
FR10. Users must not specify TIMES process set membership (e.g., PRC_GRP, ELE/DMD/etc).
The compiler assigns these based on role signatures/patterns. Minimum requirement:
  - Any role that produces a service commodity (DEM) is lowered as a demand-device pattern
    with PRC_GRP including DMD (as appropriate).

7.4 Substitutable alternatives
FR11. Multiple variants implementing the same role are compiled into multiple concrete TIMES
processes with identical role-level topology (same output service commodity), such that they
compete in the optimization. No explicit TIMES “meta-process” is required.

7.5 Deterministic symbol generation
FR12. VedaLang user-facing IDs are not TIMES symbols.
The compiler generates deterministic TIMES/VEDA symbols using a registry/abbreviation policy.
This PRD aligns with `docs/reference/vedalang-naming-matters.prd.txt`, but updates the authoring rule:
  - User does not author encoded prefixes like `C:ELC` or `S:...`.
  - Those remain compiler-generated outputs.

=====================================================================
8. Linter / Compiler Checks (No Auto-Add)
=====================================================================

FR13. The compiler/linter must enforce the following:

- Uniqueness
  - Commodity ids unique.
  - Role ids unique.
  - Variant ids unique.

- Noun/verb naming conventions (lint)
  - Commodities must be nouns (convention check; warn initially).
  - Roles must be verb-phrases (convention check; warn initially).

- Role signature validity
  - Role inputs/outputs must reference declared commodities.
  - Service-producing roles must output at least one service commodity.

- Availability completeness
  - Every variant referenced in availability must be declared.
  - Optional: warn on variants declared but never made available.

- Demand feasibility (structural)
  - For each (region, segment) demand of a service commodity, there must exist at least one
    available variant whose role can produce that commodity in that region/segment.

- Input reachability (structural)
  - For each available variant, for each required input commodity, there must exist some producer
    path in the same region (or via trade if enabled by explicit trade modules).
  - If missing: emit actionable diagnostic suggestions (do not auto-add prerequisites).

=====================================================================
9. Proposed YAML Shape (Illustrative)
=====================================================================

This is a sketch of the target user-facing syntax (exact schema subject to iteration):

model:
  name: MiniSystem1_v2
  regions: [SINGLE]
  milestone_years: [2020, 2030]

segments:
  sectors: [RES, COM]
  # end_uses: [lighting, heating]  # optional refinement

commodities:
  - id: electricity
    kind: carrier
    unit: PJ
    tradable: true
  - id: lighting
    kind: service
    unit: PJ
    tradable: false

roles:
  - id: generate_electricity
    stage: conversion
    inputs: []
    outputs: [{ commodity: electricity }]
  - id: deliver_lighting
    stage: end_use
    inputs: [{ commodity: electricity }]
    outputs: [{ commodity: lighting }]

variants:
  - id: simple_generator
    role: generate_electricity
    efficiency: 1.0
    variable_om_cost: 10
    lifetime: 40
  - id: led_lighting_device
    role: deliver_lighting
    efficiency: 0.4
    lifetime: 15
    capex: 100

availability:
  - variant: simple_generator
    regions: [SINGLE]
  - variant: led_lighting_device
    regions: [SINGLE]
    sectors: [RES, COM]

process_parameters:
  # instance-specific parameters like existing capacity and bounds
  # are bound to (variant, region, [segment]) via selectors (no user instance IDs).
  - selector:
      variant: simple_generator
      region: SINGLE
    existing_capacity:
      - vintage: 2010
        capacity: 100
    cap_bound: { up: 1000 }

demands:
  - commodity: lighting
    region: SINGLE
    sector: RES
    values: { "2020": 50 }
  - commodity: lighting
    region: SINGLE
    sector: COM
    values: { "2020": 30 }

=====================================================================
10. Implications for the Compiler (Implementation Requirements)
=====================================================================

FR14. Parser / schema updates
- Add new top-level blocks: `segments`, `roles`, `variants`, `availability`.
- Deprecate/remove (in examples first, then schema):
  - commodity `context`
  - process `sets` (user-authored)
  - process `technology` if it duplicates `variant` identity
- Replace process instance IDs with selector-based parameter blocks.

FR15. IR updates
- Extend the VedaLang AST/IR to represent roles and variants distinctly.
- Add a normalized “availability-expanded” view:
  - For each available (variant, region, segment?) create a concrete internal instance key:
    key = (variant_id, region, segment_key|None)
  - Segment key derives from `segments` configuration.

FR16. Lowering to TableIR/Excel
- Emit FI_* declarations for commodities and processes based on expanded instances.
- Emit TIMES/VEDA set membership consistent with `times/maplists.def`:
  - COM_TYPE membership for commodities derived from kind.
  - PRC_GRP membership derived from role/pattern.
- Handle non-tradable commodity scoping by segment as required.

FR17. Deterministic naming
- Implement a registry that maps user IDs to emitted abbreviations/symbols.
- Emitted symbol stability is required across compilation runs.

=====================================================================
11. Minisystem Example Migration (Required)
=====================================================================

FR18. Update minisystem examples to stop encoding semantics in names.
Example: `vedalang/examples/minisystem/minisystem1.veda.yaml` currently uses:
  - commodity name `C:ELC` (carrier) and `S:RSD:RES.ALL` (service with embedded segment)
  - process templates and instances with `technology`, `role`, and `sets`

Target migration:
- Replace encoded commodity names with nouns:
  - `C:ELC` → `electricity`
  - `S:RSD:RES.ALL` → `lighting` (or `residential_demand` if the intent is generic)
- Move sector/segment from commodity naming into the segments/availability/demand blocks.
- Replace process templates/instances with roles/variants/availability and selector-based parameters.

Acceptance criteria for minisystem updates:
- Minisystem models remain solvable (where they were solvable before).
- The compiler emits identical or strictly-more-deterministic TIMES symbols and set membership.
- The examples become readable without decoding prefixes.

=====================================================================
12. Acceptance Criteria
=====================================================================

AC1. A model author can define a RES using `roles` + `commodities` + `segments` only,
    and later add `variants` to “fill in” implementations.
AC2. For any demand, the linter gives a clear “no provider available” diagnostic that references
    the missing role/variant needed.
AC3. No user-authored TIMES set names appear in VedaLang source.
AC4. The compiler emits COM_TYPE membership aligned with TIMES `maplists.def` hard-coded sets.
AC5. Updating segmentation granularity (sector-only → sector×end_use) does not require renaming commodities.

=====================================================================
13. Open Questions
=====================================================================

OQ1. Segment model: do we standardize only on sectors first, and add end_uses later, or add both now?
OQ2. Do we allow role signatures with multiple outputs (MIMO) in the first iteration, or gate them behind tests?
OQ3. How do we represent optional “distribution/trade/storage” modules in the availability system?
OQ4. How do we reconcile or supersede process-ID geography embedding from `vedalang-naming-matters.prd.txt`
     under a “no user instance IDs” authoring model (likely: keep geography in emitted IDs only)?
